# 監視・通知・レート制限方針

本書は、システムの安定運用に必要な **監視・通知チャネル** と **レート制限ルール** を定義する。  
β 版（約 20 名）の規模に合わせ、無料枠で運用可能かつ実用的な構成とする。

---

## 1. 監視・通知の全体方針

### 設計思想

- **シンプル**: β 版では外部サービスを最小限にし、運用コストを抑える
- **確実**: 重大エラーは確実にスタッフに届く仕組みにする
- **段階的**: β 版 → 本番規模への移行時に段階的に拡張できる設計にする

### 通知チャネル

| チャネル | 用途 | コスト | β 版での採用 |
|---------|------|--------|-------------|
| **Resend メール** | S1（重大）エラーの即時通知 + 月次レポート | 無料枠: 100 通/日 | ✅ 採用 |
| **Vercel Logs** | API エラー・パフォーマンスの確認 | 無料 | ✅ 採用 |
| **Supabase Logs** | DB エラー・スロークエリの確認 | 無料 | ✅ 採用 |
| **Sentry** | エラートラッキング・パフォーマンス監視 | 無料枠: 5K イベント/月 | 🔸 任意（将来導入推奨） |
| **Slack** | リアルタイム通知 | 無料 | ❌ β 版不要 |

**β 版の結論**: **Resend メール + Vercel/Supabase のビルトインログ** の組み合わせで運用する。  
Sentry は有用だが β 版では必須ではない。安定運用後に導入を検討する。

## 2. エラー重大度と通知ルール

### 重大度定義

| レベル | 名称 | 意味 | 通知方法 |
|--------|------|------|---------|
| **S1** | 重大 | ユーザーがサービスを利用できない | **即時メール**（`ADMIN_EMAILS`） |
| **S2** | 警告 | 一部機能が劣化しているが利用可能 | **ログ記録** + 翌営業日確認 |
| **S3** | 情報 | 軽微な問題。運用に影響なし | **ログ記録のみ** |

### S1 に該当するイベント

| イベント | 説明 |
|---------|------|
| LLM 全経路失敗 | プライマリ + フォールバック両方が失敗 |
| DB 接続失敗 | Supabase への接続が確立できない |
| 月次レポート送信失敗 | Cron 実行時のレポートメール送信がエラー |
| 認証基盤障害 | Supabase Auth がダウンしてログインできない |

### S2 に該当するイベント

| イベント | 説明 |
|---------|------|
| LLM プライマリ失敗（フォールバック成功） | フォールバックで救済されたが、プライマリに問題がある |
| 画像アップロード失敗 | Storage 障害。テキストチャットは利用可能 |
| レート制限発動 | 特定ユーザーの使いすぎ |

### S3 に該当するイベント

| イベント | 説明 |
|---------|------|
| 非許可メールでのログイン試行 | 不正アクセスではなく、単に未登録の可能性が高い |
| バリデーションエラー | 画像サイズ超過など、ユーザー操作によるもの |

## 3. 通知メールのフォーマット

### S1 通知メール

```
件名: 🚨 [Marubo AI] S1: LLM 全経路失敗

━━━━━━━━━━━━━━━━━━━━
  Marubo AI 障害通知（S1 - 重大）
━━━━━━━━━━━━━━━━━━━━

■ エラー内容
  LLM API が利用できません（プライマリ + フォールバック両方失敗）

■ 発生日時
  2026-02-15 14:32:05 JST

■ リクエスト ID
  req_a1b2c3d4

■ 影響範囲
  全ユーザーのチャット機能が停止

■ 推奨アクション
  1. Vercel Logs で詳細を確認
  2. LLM プロバイダーのステータスページを確認
  3. 復旧しない場合は docs/operational/runbook.md を参照

━━━━━━━━━━━━━━━━━━━━
※ このメールは Marubo AI から自動送信されています。
```

### 通知先の環境変数

| 変数 | 用途 | 例 |
|------|------|-----|
| `ADMIN_EMAILS` | S1 通知先（本番スタッフ） | `staff1@example.com;staff2@example.com` |
| `DEV_ALERT_EMAILS` | 開発環境の通知先 | `dev@example.com` |

## 4. notifier の実装方針

```typescript
// src/shared/lib/notifier.ts

export async function notifyError(
  severity: 'S1' | 'S2' | 'S3',
  title: string,
  detail: string,
  requestId?: string
): Promise<void> {
  // S1: 即時メール送信（ADMIN_EMAILS）
  // S2: console.warn + 構造化ログ（将来 Sentry 対応可）
  // S3: console.info のみ
}
```

- S1 通知はメール送信の失敗自体を catch し、二重障害時はコンソールログに記録する（無限ループ防止）
- 同一エラーの連続通知を防ぐため、**5 分間のデバウンス**を実装する（同じ `code` のエラーは 5 分に 1 回まで）

## 5. レート制限

### 方針

β 版の規模（約 20 名）では DDoS 対策レベルのレート制限は不要だが、  
**生徒の過剰利用によるコスト暴走を防ぐ** ための月間クォータと、  
**最低限の API 保護** としての分単位レート制限を設ける。

### 月間クォータ（usage_counters）

| 項目 | 値 | 備考 |
|------|-----|------|
| **制限単位** | **ユーザー単位** | `app_user.id` ごとにカウント |
| **月間質問上限** | **100 問/月** | 環境変数 `MONTHLY_QUOTA=100` で調整可能 |
| **カウント対象** | `/api/chat` への POST（`sender = 'user'` のメッセージ） | AI の応答はカウントしない |
| **リセットタイミング** | 毎月 1 日 00:00 JST | `usage_counters.day` の月集計でリセットされる |

**超過時の動作**:
- HTTP `429 Too Many Requests` を返す
- UI 表示: 「今月の質問上限（100 回）に達しました。来月になると再びご利用いただけます。ご不明点はスタッフにお問い合わせください。」

### 分間レート制限（rate_limiter）

| 項目 | 値 | 備考 |
|------|-----|------|
| **制限単位** | **ユーザー単位** | キー: `chat:user:{user_id}` |
| **上限** | **10 リクエスト / 1 分** | 連打・ボット対策 |
| **ウィンドウ** | 固定長 1 分（`date_trunc('minute', now())`） | `rate_limiter` テーブルで管理 |

**超過時の動作**:
- HTTP `429 Too Many Requests` を返す
- UI 表示: 「送信が早すぎます。しばらく待ってからもう一度お試しください。」

### 実装フロー

```
[POST /api/chat]
  1. 認証チェック
  2. 分間レート制限チェック（rate_limiter テーブル）
     → 超過なら 429
  3. 月間クォータチェック（usage_counters テーブル集計）
     → 超過なら 429
  4. LLM 呼び出し + DB 保存
  5. usage_counters を +1 インクリメント
  6. レスポンス返却
```

### rate_limiter のクリーンアップ

- 古い `rate_limiter` レコード（`window_start < now() - interval '1 hour'`）は定期的に削除
- β 版では `/api/reports/monthly` の Cron 実行時に合わせてクリーンアップするか、手動実行

## 6. 環境変数一覧（本書関連）

| 変数 | 用途 | デフォルト | 備考 |
|------|------|-----------|------|
| `MONTHLY_QUOTA` | 月間質問上限 | `100` | `deployment.md` に記載済み |
| `ADMIN_EMAILS` | S1 通知先 | — | `;` 区切り |
| `DEV_ALERT_EMAILS` | 開発通知先 | — | 任意 |
| `RESEND_API_KEY` | メール送信 | — | Resend ダッシュボードで取得 |
| `MAIL_FROM` | 送信元アドレス | — | 検証済みドメインのアドレス |
| `SENTRY_DSN` | Sentry（任意） | 空 | 空なら Sentry 無効 |

---

## 7. 将来の拡張（β 版対象外）

- **Sentry 統合**: `SENTRY_DSN` を設定するだけで有効化できるようにコードを準備
- **Slack 通知**: Incoming Webhook で S1 通知を Slack にも飛ばす
- **ダッシュボード**: Vercel Analytics + カスタムメトリクスで利用状況を可視化
- **IP ベースのレート制限**: 認証前の DDoS 対策（β 版では Vercel の標準保護で十分）
- **動的クォータ**: スタッフが管理画面で生徒ごとにクォータを変更できる機能

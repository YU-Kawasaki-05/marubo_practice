## 1. このプロジェクトは何か

### ひとことで言うと

**学習塾の生徒が AI に質問できるチャットボット Web アプリ**です。

### もう少し詳しく

- **生徒**がブラウザからテキストや画像で質問を送ると、**AI（LLM）が回答**します
- 回答は**数式（LaTeX/KaTeX）やマークダウン**に対応しています（例: `$x^2 + y^2 = r^2$` のような数式がきれいに表示される）
- 会話は**データベースに保存**され、生徒は自分の過去の会話を見返せます
- **塾のスタッフ（管理者）**は、全生徒の会話を検索・閲覧できます
- 毎月、**月次レポート（CSV/HTML）**がスタッフのメールに自動送信されます
- 現在は**β版**で、約20人程度の小規模利用を想定しています

### 登場人物（ロール）

| ロール | できること |
|--------|------------|
| **生徒（student）** | ログイン → テキスト/画像で質問 → AI回答を見る → 会話履歴を見返す |
| **スタッフ（staff）** | ログイン → 全生徒の会話検索/閲覧 → 許可メール管理 → 月次レポート受信/再実行 |

### 利用の流れ（ユーザーストーリー）

```
1. スタッフが生徒のGmailを「許可メールリスト」に登録する
2. 生徒がGoogleアカウントでログインする
3. サーバーが「この生徒は許可リストにいるか？」をチェックする
4. 許可されていれば、チャット画面で質問ができるようになる
5. 生徒が質問を送ると、AIが回答を返す
6. 会話はデータベースに保存される
7. スタッフは管理画面で会話を検索・閲覧できる
8. 月末に自動で月次レポートがメール送信される
```

## 7. 詳細タスク分解（LLMへの指示用）

> **利用方法**: 以下のタスクは番号順に実行することを推奨します。
> 各タスクは「前提条件」「やること」「完了条件」が明記されているので、
> 能力が高くない LLM でも正確に実行できるように設計しています。

---

### タスク 1: conversation / message テーブルのマイグレーション SQL 作成

**前提条件**: なし（最初にやるタスク）

**背景**:
チャット機能を作るには、会話データを保存するテーブルが必要です。
`docs/database.md` に設計が記載されています。

**やること**:

1. `supabase/migrations/` ディレクトリに新しい SQL ファイルを作成する
   - ファイル名: `20260215000000_chat_tables.sql`
2. 以下のテーブルを作成する SQL を書く:

```sql
-- conversation テーブル
CREATE TABLE IF NOT EXISTS conversation (
  id         UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id    UUID NOT NULL REFERENCES app_user(id) ON DELETE CASCADE,
  title      TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

-- message テーブル
CREATE TABLE IF NOT EXISTS message (
  id              UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  conversation_id UUID NOT NULL REFERENCES conversation(id) ON DELETE CASCADE,
  role            TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system')),
  content         TEXT NOT NULL,
  token_count     INTEGER,
  created_at      TIMESTAMPTZ DEFAULT now()
);

-- インデックス
CREATE INDEX idx_conversation_user_created ON conversation(user_id, created_at DESC);
CREATE INDEX idx_message_conv_created ON message(conversation_id, created_at);
```

3. RLS ポリシーを設定する:

```sql
-- conversation の RLS
ALTER TABLE conversation ENABLE ROW LEVEL SECURITY;

-- 生徒: 自分の会話のみ読み取り
CREATE POLICY "student_select_own_conversations"
  ON conversation FOR SELECT
  USING (user_id = (SELECT id FROM app_user WHERE auth_uid = auth.uid()));

-- スタッフ: 全件読み取り
CREATE POLICY "staff_select_all_conversations"
  ON conversation FOR SELECT
  USING (auth.jwt() -> 'app_metadata' ->> 'role' = 'staff');

-- message の RLS
ALTER TABLE message ENABLE ROW LEVEL SECURITY;

-- 生徒: 自分の会話のメッセージのみ
CREATE POLICY "student_select_own_messages"
  ON message FOR SELECT
  USING (conversation_id IN (
    SELECT id FROM conversation
    WHERE user_id = (SELECT id FROM app_user WHERE auth_uid = auth.uid())
  ));

-- スタッフ: 全件
CREATE POLICY "staff_select_all_messages"
  ON message FOR SELECT
  USING (auth.jwt() -> 'app_metadata' ->> 'role' = 'staff');
```

4. `src/shared/types/database.ts` に型定義を追加する:

```typescript
// 既存の型定義に以下を追加

export type ConversationRow = {
  id: string
  user_id: string
  title: string | null
  created_at: string
  updated_at: string
}

export type MessageRow = {
```

## 9. 開発を進める際の注意点

### コーディングルール

1. **TypeScript strict モード**: `any` を使わない。型を明示的に定義する
2. **パスエイリアス**: `@features/*`, `@shared/*` を使う（tsconfig.json で設定済み）
3. **コンポーネント命名**: PascalCase（例: `ChatInput.tsx`）
4. **関数命名**: camelCase（例: `sendMessage.ts`）
5. **Conventional Commits**: `feat(chat): メッセージ送信機能を追加` の形式

### セキュリティルール

1. **Service Role は必ずサーバーサイド（API Route）でのみ使用する**。ブラウザに漏らさない
2. **RLS ポリシーを必ず設定する**。新しいテーブルには必ず RLS を有効にする
3. **rehype-sanitize** でマークダウンを無害化する（XSS 対策）
4. **環境変数は `.env.local` に置き、`.env.example` にはダミー値のみ記載する**

### テストルール

1. `pnpm typecheck` → `pnpm lint` → `pnpm test` の順で確認
2. テストは日本語の `describe` ブロックで書く（既存テストに合わせる）
3. Supabase 関連のテストは `MOCK_SUPABASE=true` で動く mock を使う

### 品質チェックコマンド（必ず実行）

```bash
pnpm typecheck   # TypeScript の型チェック
pnpm lint        # ESLint によるコードチェック
pnpm test        # Vitest によるテスト実行
pnpm build       # Next.js のビルド（本番用）
```

## 10. 用語集

| 用語 | 説明 |
|------|------|
| **Next.js** | React ベースの Web フレームワーク。フロントエンドとバックエンドを 1 プロジェクトで作れる |
| **App Router** | Next.js 14 以降の新しいルーティング方式。`app/` ディレクトリにファイルを置くと自動でページになる |
| **Route Handler** | `app/api/xxx/route.ts` に書く API エンドポイント。サーバーサイドで実行される |
| **Supabase** | Firebase の代替となるオープンソース BaaS。Postgres, Auth, Storage を提供 |
| **RLS (Row Level Security)** | Postgres の行レベルセキュリティ。「この行は誰が見てよいか」をDB側で制御する |
| **Service Role** | Supabase の管理者権限キー。RLS をバイパスできるため、サーバー側のみで使用する |
| **Bearer トークン** | HTTP リクエストの `Authorization: Bearer xxx` ヘッダで送るアクセストークン |
| **JWT** | JSON Web Token。ログインセッションの情報をトークンとして持つ |
| **LLM** | Large Language Model（大規模言語モデル）。ChatGPT のような AI |
| **KaTeX** | ブラウザで数式を表示するためのライブラリ |
| **Markdown** | テキストの装飾記法（太字、リスト、コードブロック等） |
| **Migration** | データベースの構造変更を SQL ファイルで管理する仕組み |
| **Vitest** | テスト実行ツール（Jest の高速版） |
| **ESLint** | JavaScript/TypeScript のコード品質チェックツール |
| **Prettier** | コードの自動整形ツール |
| **Conventional Commits** | `feat:`, `fix:`, `docs:` のようなプレフィックスでコミットメッセージを書く規約 |
| **Cron** | 定期実行ジョブ。このプロジェクトでは Vercel Cron で毎日 23:55 JST に実行 |
| **Resend** | メール送信 API サービス |
| **CSP (Content Security Policy)** | XSS 攻撃を防ぐための HTTP ヘッダ |
| **Zustand** | React の状態管理ライブラリ（Redux より軽量） |
| **pnpm** | npm の代替パッケージマネージャー（高速・ディスク効率がよい） |

# 12. セキュリティ評価

### 実装されているセキュリティ対策

| カテゴリ | 対策 | 詳細 |
|---|---|---|
| 認証 | Supabase Auth | メール/パスワード認証、JWTトークン |
| 認可 | RLS | 全データテーブルにRow-Level Securityを適用 |
| 認可 | requireStaff() | 管理系APIでスタッフロールを必須チェック |
| 認可 | GRANT_ALLOWED_EMAILS | 権限管理操作の二重認可 |
| XSS対策 | rehype-sanitize | Markdownレンダリング時のHTML無害化 |
| CSP | Content-Security-Policy | script-src, style-src等を制限 |
| HTTPヘッダ | セキュリティヘッダ群 | HSTS, X-Frame-Options, X-Content-Type-Options |
| 権限管理 | Permissions-Policy | カメラ・マイク・位置情報を無効化 |
| 入力検証 | サーバー側バリデーション | MIME タイプ・ファイルサイズの再検証 |
| 監査 | 監査ログ | 許可リスト操作・権限変更を全記録 |
| データ保護 | メール正規化 | 小文字化・トリムで一貫性確保 |

### 潜在的なセキュリティリスクと推奨対策

| リスク | 詳細 | 推奨対策 |
|---|---|---|
| エラー詳細の露出 | `POST /api/chat` がエラー詳細をクライアントに返す（コメントに「本番では隠すべき」と記載あり） | 本番環境ではエラー詳細を隠蔽し、汎用エラーメッセージを返す |
| デバッグログの残存 | `GET /api/conversations` にJWTのiss/aud/expをログ出力するコードが残っている | 本番デプロイ前にデバッグログを削除する |
| APIレート制限未実装 | OpenAI APIコール等に対するレート制限が実装されていない | API Gatewayまたはミドルウェアでレート制限を実装する |
| CSRF対策 | 明示的なCSRF対策が確認されない（Bearer認証のためリスクは低い） | SameSite Cookie設定の確認 |
| Service Role Key | サーバーサイドでService Role Keyを使用している | 環境変数の適切な管理、定期的なキーローテーション |
